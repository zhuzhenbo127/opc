# 资产沉淀

## 目标

帮助用户判断“哪些成果值得沉淀为资产”，而不是替用户一次性搭完整系统。

## 核心原则

- 默认读写当前工作目录下的 `opc-doc/`
- 教学模式下先解释“资产沉淀”和“复利”
- 默认一次只问一个问题；如果几个问题都很轻、彼此紧密相关，可以合并成 2 到 3 个
- 默认给 3 种优先沉淀方向，并附加 `4. 我有自己的方案`
- 用户确认后再写入正式结果
- 不直接给推荐结论，只做方案分析
- **本阶段做资产优先级判断，不替用户直接生产资产内容**

## 本阶段边界

### 本步做什么

- 识别哪些成果值得沉淀为资产（判断，不是执行）
- 明确资产类别和优先沉淀顺序
- 形成资产沉淀方案（规划层）

### 本步不做什么

- ❌ 不替用户直接写内容（如文案、帖子、模板正文）——除非用户明确要求
- ❌ 不做完整内容日历排期（可以规划类别，不展开每一条）

**注意**：资产沉淀阶段是整套方法论中最靠近执行的一步。允许输出具体的模板结构框架，但"填写内容"由用户自己完成，或在用户明确要求后才协助。

## 本步骤必须完成什么

1. 识别可沉淀成果
2. 明确资产类别
3. 明确优先沉淀方向
4. 形成当前资产沉淀方案

## 优先确认顺序

1. 哪些成果是重复会用到的
2. 这些成果属于哪类资产
3. 当前先沉淀哪一类
4. 这类资产如何组织

## 完成标准

- 已形成 3 种资产沉淀优先方案
- 用户已确认当前优先沉淀方向

## 本步需要解释什么

教学模式下先解释：

- 资产沉淀不是记笔记，而是把未来还会用的东西整理出来
- 资产可以是内容、方法、渠道、数据或品牌信任
- 这一步的作用是让业务越来越不依赖临场重复劳动

## 触发条件

**本技能不是建盘期的线性下一步，而是运营循环中的条件触发工具。**

触发时机：用户在实际运营中发现某些动作开始重复出现，想把它们系统化，不再每次从零做起。

常见触发信号：
- "我已经写了很多类似的内容，感觉可以整理成模板"
- "这套流程我用了好几次了，应该固化下来"
- "客户问的问题都差不多，我想做个标准回答"

**前置条件检查**：如果用户还没有实际运营产出（完全没做过），不进入本技能，建议先去执行、再回来。

## 输入

优先读取：

- `opc-doc/outputs/06-mvp-design/*`
- `opc-doc/outputs/07-conversion-loop/*`
- `opc-doc/outputs/08-asset-ops/`（历史沉淀记录，本次是否有新增）

如果上述文件缺失，判断当前对话中是否已有可复用成果描述。

- 如果没有，说明当前还没有可沉淀的产物，建议先去运营一段时间再回来

## 执行步骤

1. 解释本步目标
2. 默认一次只问一个问题；如果几个问题都很轻、彼此紧密相关，可以合并成 2 到 3 个，例如：
   - 你最近重复做的动作是什么？
   - 哪类内容或经验你未来一定还会再用？
   - 你更想先沉淀方法、内容还是客户线索？
3. 每轮回答后，给简短反馈
4. 生成 3 种优先沉淀方案，例如：
   - 方法模板优先
   - 内容资产优先
   - 渠道与线索资产优先
5. 说明每种方案的适用情况、优点和代价
6. 默认增加 `4. 我有自己的方案`
7. 让用户确认、组合、修改，或直接提出自己的版本
8. 用户确认后，再写入正式结果

## 输出

对话层必须包含：

1. 本步解释
2. 当前可沉淀成果摘要
3. 3 种资产沉淀优先方案 + `4. 我有自己的方案`
4. 每个方案的适用情况、优点和代价
5. 请用户确认或修改

## 落盘检查点（每次触发后必须完成）

用户明确确认资产沉淀优先方向后，**立即**使用 Write 工具写入文件。在对话中描述结论不等于落盘。

**本技能支持多次触发。每次落盘使用带日期的文件名，不覆盖历史记录。**

**写入文件（使用当天日期，格式 YYYYMMDD）：**

- `opc-doc/outputs/08-asset-ops/asset-inventory-[YYYYMMDD].md`（本次可沉淀成果清单 + 资产类别标注）
- `opc-doc/outputs/08-asset-ops/action-plan-[YYYYMMDD].md`（本次确认的优先沉淀方向和下一步动作）

**更新状态文件：**

- `opc-doc/state/current-stage.json`（写入：`{"stage": "08-asset-ops", "status": "completed", "next_stage": "execution", "summary": "一句话本次优先沉淀方向"}`）
- `opc-doc/state/decisions.json`（追加本次资产沉淀优先级决策，注明日期）

**落盘完成后，在对话中告知用户：**
> "✅ 本次资产沉淀方案已保存到 opc-doc/。继续去执行——当你运营遇到瓶颈时，可以触发经营复盘（`opc-dashboard-review`）。"

## 何时调用其他 skills

- 本技能完成后，不强制推进到其他技能
- 如果用户同时反映有运营瓶颈，可建议触发 `opc-dashboard-review`
- 如果本次沉淀发现前置策略有问题，可建议回退到对应建盘阶段

## 异常处理

- 如果用户还没有稳定产物，不进入正式资产沉淀
- 如果产物很多但很散，先做分类，不急着做完整体系
